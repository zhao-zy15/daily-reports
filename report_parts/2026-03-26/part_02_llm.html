TuneShift-KD: Knowledge Distillation and Transfer for Fine-tuned Models

Yushi Guan, Jeanine Ohene-Agyei, Daniel Kwan, Jean Sebastien Dandurand, Yifei Zhang, Nandita Vijaykumar · PDF · cs.LG · 2026-03-25
📌 背景与动机

在实际部署中，企业常用 LoRA 等参数高效微调技术将专业知识注入基础模型。然而，当新架构（如从 LLaMA-2 迁移到 LLaMA-3）出现时，原始微调数据往往因隐私或商业限制不可获取。现有的知识蒸馏方法要么需要完整的原始训练数据，要么需要训练额外的判别器，成本高昂且不够灵活。TuneShift-KD 旨在解决无原始数据条件下，将微调模型的专业知识自动蒸馏到新架构模型中的问题。

🔬 核心方法详解
① 困惑度差异驱动的知识识别 (Perplexity-Gap Knowledge Detection)

核心洞察：专业知识可通过基础模型与微调模型的困惑度差异来定位。给定提示 \(x\) 和微调模型响应 \(y\)，定义知识指标：

$$\Delta_{\text{PPL}}(x, y) = \text{PPL}_{\text{base}}(y | x) - \text{PPL}_{\text{ft}}(y | x)$$

当 \(\Delta_{\text{PPL}}\) 较大时，说明微调模型对该响应"自信"而基础模型"困惑"，意味着这是微调过程中学到的专门知识。通过设定阈值 \(\tau\) 筛选出高知识密度的样本对 \(\{(x_i, y_i) \mid \Delta_{\text{PPL}}(x_i, y_i) > \tau\}\)。

② 迭代合成数据生成 (Iterative Synthetic Data Generation)

初始只需少量（~50 个）代表性提示。通过以下迭代过程扩充训练数据：

  1. 提示扩展：利用微调模型生成与高 \(\Delta_{\text{PPL}}\) 提示语义相似的新提示 \(x' \sim p_{\text{ft}}(\cdot | x_{\text{seed}})\)
  2. 响应生成：对每个新提示用微调模型生成响应 \(y' = \arg\max p_{\text{ft}}(y | x')\)
  3. 过滤：仅保留 \(\Delta_{\text{PPL}}(x', y') > \tau\) 的样本，每轮迭代数据量约增长 2-3 倍

经 3-5 轮迭代，可生成数千条高质量合成蒸馏数据，无需访问任何原始训练数据。

③ 跨架构知识蒸馏 (Cross-Architecture Distillation)

使用生成的合成数据集 \(\mathcal{D}_{\text{syn}}\)，通过标准 KL 散度损失训练目标模型：

$$\mathcal{L}_{\text{KD}} = \sum_{(x,y) \in \mathcal{D}_{\text{syn}}} \text{KL}\left(p_{\text{ft}}(y|x) \| p_{\theta}(y|x)\right) + \lambda \cdot \mathcal{L}_{\text{CE}}(y, p_{\theta}(y|x))$$

其中 \(p_{\theta}\) 为新架构的目标模型，\(\lambda\) 平衡蒸馏与交叉熵损失。关键在于蒸馏数据已聚焦于"专业知识"区域，因此即便数据量较少也能高效迁移。

④ 自动化无判别器流程 (Discriminator-Free Automation)

与以往需要训练 GAN 风格判别器来判断知识质量的方法不同，TuneShift-KD 完全基于困惑度信号，实现端到端自动化：

  • 无需额外的判别器训练开销
  • 只需基础模型 + 微调模型 + 少量种子提示
  • 整个过程的计算成本约为从头微调的 15-20%
💡 深度案例
🔍 案例：医疗领域 LoRA 迁移 (LLaMA-2-7B → LLaMA-3-8B)

场景：某医疗机构用 5000 条医患对话在 LLaMA-2-7B 上做了 LoRA 微调，现需迁移到 LLaMA-3-8B，但原始数据因 HIPAA 合规不可访问。

  • 种子提示：50 条通用医疗咨询模板（如"患者出现持续性头痛和视力模糊，请问可能的诊断是？"）
  • 迭代 3 轮后：生成 2,847 条合成专业数据，其中 \(\Delta_{\text{PPL}} > 15.3\) 的样本占 68%
  • 蒸馏结果：目标模型在 MedQA 基准上准确率达到 71.2%，而直接用 LLaMA-3-8B 基础模型仅 54.8%，恢复了源微调模型 73.5% 精度的 96.9%

Schema on the Inside: A Two-Phase Fine-Tuning Method for High-Efficiency Text-to-SQL at Scale

Chinmay Soni, Shivam Chourasia, Gaurav Kumar, Hitesh Kapoor · PDF · AAAI-26 · cs.CL, cs.AI · 2026-03-25
📌 背景与动机

将大型专有 API 模型（如 GPT-4、Gemini）应用于 Text-to-SQL 时，必须在提示中包含完整数据库 Schema（通常 15K-20K tokens），导致 API 费用高昂和延迟不可接受。对于拥有 2.5 亿用户的 CriQ（Dream11 旗下板球幻想体育平台），每次查询的 API 开销使生产部署不可行。本文提出让 8B 参数自托管模型"内化"整个数据库 Schema，从而将输入 token 数从 17K 降至不到 100。

🔬 核心方法详解
① 阶段一：Schema 内化预训练 (Schema Internalization Pre-training)

在 8B 模型上进行持续预训练，让模型"记住"完整的数据库结构：

  • Schema 文档化：将数据库的表名、列名、数据类型、外键关系、值域约束转化为结构化自然语言描述
  • 训练数据：生成 Schema Q&A 对（如"板球比赛表有哪些列？" → 完整列枚举）、Schema 导航任务（给定查询意图，选择相关表和列）
  • 损失函数：$$\mathcal{L}_{\text{schema}} = -\sum_{t=1}^{T} \log p_\theta(s_t | s_{

经过此阶段，模型在没有任何 Schema 注入的情况下即可回答关于数据库结构的问题。

② 阶段二：SQL 生成监督微调 (SQL Generation SFT)

在模型已内化 Schema 的基础上，用（自然语言查询, SQL）对进行标准 SFT：

$$\mathcal{L}_{\text{SQL}} = -\sum_{t=1}^{T} \log p_\theta(\text{SQL}_t | \text{SQL}_{关键区别：输入中完全不包含 Schema 信息，模型必须从内化的知识中检索所需的表和列。这使得推理时的输入仅为用户的自然语言问题（< 100 tokens），而非传统方法的 Schema + 问题（~17K tokens）。

③ 99% Token 压缩率的推理优化 (Inference Optimization)

传统 RAG/Prompt 方案 vs Schema-on-the-Inside：

  • 传统方案：每次推理 = Schema (15K-17K tokens) + 查询 (~100 tokens) → API 调用
  • 本方案：每次推理 = 查询 (~100 tokens) → 本地 8B 模型推理
  • Token 减少率：\(\frac{17000 - 100}{17000} \approx 99.4\%\)

在 A100 GPU 上，单次推理延迟从 API 的 2-5 秒降至本地的 0.3 秒。

④ 执行验证与自修正循环 (Execution Verification Loop)

生成的 SQL 通过执行验证流程：如果 SQL 执行出错或返回空结果，触发自修正：模型接收错误信息后重新生成，最多 3 次迭代。这将执行成功率从单次生成的 94.1% 提升至 98.4%。

MARCH: Multi-Agent Reinforced Self-Check for LLM Hallucination

Zhuo Li, Yupeng Zhang, Pengyu Cheng, Jiajun Song, Mengyu Zhou, Hao Li, Shujie Hu, Yu Qin 等 · PDF · Code · cs.CL · 2026-03-25
📌 背景与动机

RAG 系统中的幻觉检测通常采用"LLM-as-a-judge"方式——用同一个或另一个 LLM 根据检索到的证据验证生成内容。但这种方法存在根深蒂固的确认偏差 (confirmation bias)：验证者能看到原始生成的输出，往往会无意中"复读"和"认可"原始错误。MARCH 通过精心设计的信息不对称 (information asymmetry)，打破这一自我确认循环。

🔬 核心方法详解
① 三角色多智能体架构 (Three-Role Multi-Agent Architecture)

MARCH 定义三个独立角色，各自拥有不同的信息访问权限：

  • Solver（求解者）：接收用户 query \(q\) 和检索文档 \(\mathcal{D}\)，生成初始 RAG 响应 \(r = f_{\text{solve}}(q, \mathcal{D})\)
  • Proposer（提议者）：将响应 \(r\) 分解为原子命题集 \(\{c_1, c_2, \ldots, c_n\} = f_{\text{propose}}(r)\)，每个 \(c_i\) 为可独立验证的最小事实单元
  • Checker（检查者）：关键设计——Checker 无法看到原始响应 \(r\)，仅根据检索文档 \(\mathcal{D}\) 验证每个命题 \(v_i = f_{\text{check}}(c_i, \mathcal{D})\)，输出 \(v_i \in \{\text{SUPPORT}, \text{REFUTE}, \text{NOT\_ENOUGH\_INFO}\}\)

信息不对称公式化表示：\(\text{Info}(\text{Checker}) = \{\mathcal{D}, c_i\} \subsetneq \{\mathcal{D}, q, r, c_i\} = \text{Info}(\text{Solver})\)

② 多智能体强化学习训练 (Multi-Agent RL Training)

三个角色通过 MARL 联合训练，共同优化事实遵守能力。定义联合奖励函数：

$$R = \alpha \cdot R_{\text{factual}} + \beta \cdot R_{\text{coverage}} + \gamma \cdot R_{\text{precision}}$$

其中：

  • \(R_{\text{factual}}\)：Checker 验证通过的命题比例（鼓励 Solver 生成有据可查的内容）
  • \(R_{\text{coverage}}\)：Proposer 分解的命题覆盖原始响应信息的完整度
  • \(R_{\text{precision}}\)：Checker 判断的准确度（与人工标注对比）

使用 PPO 算法联合更新三个角色的策略，通过中心化训练 + 去中心化执行（CTDE）保证训练稳定性。

③ 原子命题分解策略 (Atomic Claim Decomposition)

Proposer 的分解需满足三个约束：

  1. 原子性：每个命题包含且仅包含一个可验证事实，避免复合命题
  2. 完整性：所有命题的并集应覆盖原始响应的全部事实内容
  3. 独立性：命题之间不应有逻辑依赖（验证 \(c_i\) 不需要知道 \(c_j\)）

通过 RL 奖励中的 \(R_{\text{coverage}}\) 项自动学习最优分解粒度，无需手动规则。

④ 信息隔离下的幻觉修正 (Hallucination Correction under Isolation)

当 Checker 判定某命题 \(c_i\) 为 REFUTE 时，触发修正流程：

  1. Solver 收到反馈 \(\{c_i: \text{REFUTE}\}\)，但不知道 Checker 的推理过程
  2. Solver 重新生成该命题对应的内容段落 \(r'_i = f_{\text{solve}}(q, \mathcal{D}, c_i, \text{REFUTE})\)
  3. 新内容再次通过 Proposer → Checker 流程验证，最多迭代 3 次

这种闭环修正使最终响应的事实准确率从单次生成的 78.3% 提升至 91.6%。

💡 深度案例
🔍 案例：RAG 幻觉检测与修正

用户问题："量子计算机何时能破解 RSA-2048 加密？"

检索文档：包含 Shor 算法原理和 IBM 量子路线图的 3 篇文献

Solver 初始响应（含幻觉）："根据 IBM 2024 年路线图，10 万量子比特的处理器预计 2029 年面世，届时可运行 Shor 算法破解 RSA-2048..."

Proposer 分解：

  • \(c_1\)：IBM 路线图提到 2029 年 10 万量子比特处理器 → Checker: REFUTE（文档仅提到 10 万物理量子比特，非逻辑量子比特，且时间线为 2033 年）
  • \(c_2\)：Shor 算法可破解 RSA-2048 → Checker: SUPPORT
  • \(c_3\)：需要约 2000 万物理量子比特才能实现破解 → Checker: SUPPORT

修正后响应："...IBM 路线图计划于 2033 年实现 10 万物理量子比特处理器，但破解 RSA-2048 需约 2000 万物理量子比特..."
