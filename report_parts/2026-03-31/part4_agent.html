AgentSwing: Golf Swing Coaching via LLM-Based Multi-Modal Agent

👤 Xiang Ren, Yingzhi Wu et al.
📅 2026-03-28
🏛 Tongyi Lab, Alibaba Group
Agent · 运动分析 · 多模态
AgentSwing 是一个基于 LLM 的多模态智能体系统，用于高尔夫挥杆教学。该系统将视频分析、姿态估计、物理模拟和自然语言对话融合在一个统一的 Agent 框架中。核心创新在于设计了一个运动知识图谱 (SwingKG) 作为 Agent 的长期记忆，存储标准挥杆模板和常见错误模式。在用户研究中，76% 的高尔夫初学者认为 AgentSwing 的反馈与职业教练同等甚至更好。

🔬 核心方法详解

① SwingKG 运动知识图谱

构建包含 5 个层级的高尔夫挥杆知识图谱：

  • 阶段层：Address → Backswing → Downswing → Impact → Follow-through（5 个阶段节点）
  • 关节层：17 个人体关键点的标准角度范围（如肩关节旋转 85-95°）
  • 力学层：力矩传递链 \(\tau_{\text{hip}} \to \tau_{\text{torso}} \to \tau_{\text{shoulder}} \to \tau_{\text{arm}}\)
  • 错误层：42 种常见错误模式及其根因分析链
  • 纠正层：每种错误对应的练习方案和注意事项

KG 由 3 位 PGA 认证教练标注，并通过运动生物力学论文交叉验证。

② 多模态分析管线

Agent 对用户上传的挥杆视频执行自动化分析链：

  1. 姿态估计：使用 ViTPose-H 从视频帧提取 17 个 2D 关键点 + 深度估计得到 3D 关键点
  2. 阶段分割：基于关键点速度曲线自动分割挥杆阶段 \(t_{\text{split}} = \arg\max_t |\dot{\mathbf{p}}_{\text{wrist}}(t)|\)
  3. 角度计算：计算每个阶段的关键关节角度和运动学参数
  4. 偏差检测：与 SwingKG 中的标准范围对比，生成偏差报告 \(\Delta\theta_j = \theta_j^{\text{user}} - \theta_j^{\text{ref}}\)
③ LLM Agent 的规划与对话

Agent 使用 ReAct 框架进行多轮对话式教学：

  • 工具调用：Agent 可调用 7 种工具（视频分析、KG 查询、物理模拟、对比可视化、练习推荐、进度跟踪、慢动作回放）
  • 记忆机制：维护用户画像（水平、常见错误、改善历史），实现个性化教学
  • 教学策略：根据用户水平自适应选择解释深度——初学者用比喻（"想象你在甩毛巾"），高级者用生物力学术语
④ 物理模拟与反事实分析

集成 MuJoCo 物理引擎，模拟关节角度变化对击球结果的影响：

$$\Delta d = f_{\text{sim}}(\Delta\theta_{\text{hip}}, \Delta\theta_{\text{shoulder}}, v_{\text{club}}, \alpha_{\text{attack}})$$

支持"如果你的肩旋转增加 10°，球会飞远 15 码"这样的反事实推理。可视化为叠加的轨迹动画，让用户直观理解纠正效果。

💡 深度案例分析
  1. 级联错误发现：用户报告"球总是向右飞"，Agent 分析发现根因不在手臂而在髋关节旋转不足（实际 62° vs 标准 85°），导致肩膀补偿性过度外旋 → 杆面开放 → 球右曲
  2. 个性化教学：同一个"右曲球"问题，对初学者说"挥杆时想象左侧口袋面向目标"，对高级者说"增加髋关节前导旋转角度至 80°，注意力矩传递序列"
  3. 持续改善追踪：3 周的纵向用户研究显示，使用 AgentSwing 的初学者平均差点从 28 降至 22，而对照组（YouTube 教程）从 28 降至 26

🔑 启示：领域知识图谱 + 多模态分析 + 物理模拟的三元组合，使 Agent 能从"描述现象"升级为"解释因果并指导改善"。

Marco DeepResearch: An Agentic System for Scientific Literature Research

👤 Lei Huang, Zhenghan Chen et al.
📅 2026-03-28
🏛 Alibaba International Digital Commerce
Agent · 科研自动化 · 文献综述
Marco DeepResearch 是一个面向科研的自主智能体系统，能够自动完成从研究问题理解 → 文献检索 → 论文精读 → 综述撰写的完整科研流程。系统采用递归分解架构——将复杂研究问题拆解为子问题树，每个叶节点由专门的子 Agent 处理。在 SciReview 基准上，Marco 生成的文献综述在覆盖度、准确性和结构质量上显著超越 GPT-4 直接生成，接近人类研究者水平。

🔬 核心方法详解

① 递归问题分解 (RPD)

输入研究问题 \(Q\) 后，系统构建问题分解树 \(\mathcal{T}\)：

  1. 根节点：原始研究问题（如"LLM 幻觉的成因与缓解方法"）
  2. 一级分解：Planner Agent 生成 3-5 个子问题（定义/分类、成因分析、检测方法、缓解策略、评估基准）
  3. 二级分解：每个子问题进一步分解为可直接检索的原子查询（如"训练数据记忆导致的幻觉"）
  4. 深度限制：最多 3 层分解，叶节点数量控制在 15-25 个

分解质量由 Critic Agent 评估，若覆盖度不足则触发追加分解。

② 多源检索与去重

每个叶节点查询由 Retriever Agent 执行多源检索：

  • Semantic Scholar API：获取语义相关论文（按引用数排序）
  • arXiv 全文搜索：获取最新预印本
  • Google Scholar：补充覆盖度

去重策略：基于标题的编辑距离 + DOI 精确匹配，重复论文保留引用数最高的版本。每个叶节点检索 20-30 篇，经相关性打分后保留 Top-5：

$$\text{Rel}(p, q) = 0.6 \cdot \text{cos}(\mathbf{e}_p, \mathbf{e}_q) + 0.3 \cdot \text{CitScore}(p) + 0.1 \cdot \text{Recency}(p)$$

③ 深度阅读与信息抽取

Reader Agent 对每篇保留的论文执行结构化阅读，抽取 7 类信息：

  • 问题定义、方法概述、关键创新、实验设置、主要结果、局限性、与查询的关联度

对 PDF 全文使用滑动窗口阅读（4K token 窗口，1K 重叠），通过多轮 QA 确保关键信息不遗漏。每篇论文生成约 500 词的结构化摘要。

④ 层次化综述生成

Writer Agent 按问题树的层次结构自底向上生成综述：

  1. 叶节点：每个原子查询生成 1-2 段的微综述
  2. 中间节点：合并子节点微综述，消除冗余，建立交叉引用
  3. 根节点：生成引言和结论，插入全局主题转换句

使用"引用链"机制确保每个论述点都有明确的文献支撑：\(\text{[Claim]} \xrightarrow{\text{supported by}} \text{[Paper1, Paper2]}\)。

💡 深度案例分析
  1. 问题分解示例："Scaling Laws for LLMs"被分解为：①定义与发展脉络 ②Chinchilla 法则及挑战 ③超越幂律的新发现 ④在下游任务的适用性 ⑤多模态 Scaling Laws——覆盖了该领域所有核心子话题
  2. 自动发现研究空白：在综合子主题综述后，Writer Agent 自动标注了"目前缺少混合精度训练对 Scaling Law 影响的系统研究"——这与该领域真实的研究空白一致

🔑 启示：递归分解是解决复杂知识任务的通用范式——它让 Agent 在每个层级只需处理可控复杂度的子问题。

GAAMA: Generalizable Adaptive Autonomous Multi-Agent System

👤 Atharva Rajkumar Lele, Mihir Rajkumar Lele
📅 2026-03-28
🏛 Nagarro
Agent · 多智能体 · 自适应
GAAMA 提出了一个通用自适应多智能体系统框架，核心创新在于动态角色分配和能力感知路由机制。传统多 Agent 系统中角色是预定义的，GAAMA 则让 Agent 根据任务需求动态组建团队：自动确定需要哪些角色、每个角色的能力要求、以及最优的协作拓扑。在 WebArena、ALFWorld 和 SWE-Bench 三个基准上，GAAMA 的动态团队比固定角色团队平均提升 12.4% 的任务完成率。

🔬 核心方法详解

① 任务分析与角色生成

Orchestrator Agent 接收任务后进行三步分析：

  1. 复杂度评估：估算任务所需的能力种类数 \(K\) 和并行度 \(P\)
  2. 角色蓝图生成：为每个所需能力生成角色描述 \(R_k = (\text{name}, \text{capability}, \text{tools}, \text{constraints})\)
  3. 拓扑选择：根据子任务依赖关系选择协作拓扑（链式、树形、网状、混合）

例如，"创建一个带数据库的 Web 应用"会生成：前端开发者、后端开发者、DBA、测试工程师 4 个角色，拓扑为"前后端并行 → 集成 → 测试"。

② 能力感知路由

设计能力匹配分数来分配子任务：

$$\text{Match}(R_k, T_j) = \sum_{c \in \text{caps}(T_j)} w_c \cdot \mathbb{1}[c \in \text{caps}(R_k)] \cdot \text{prof}(R_k, c)$$

其中 \(\text{prof}(R_k, c)\) 为角色 \(R_k\) 在能力 \(c\) 上的熟练度（通过历史表现估计）。路由问题通过匈牙利算法求解最优匹配。当任务执行失败时，触发重路由：将失败任务交给能力次优的 Agent 或生成新的专家角色。

③ 自适应协作协议

三种协作模式根据任务进展自动切换：

  • 并行模式：独立子任务同时执行，共享黑板 (Blackboard) 通信
  • 流水线模式：有依赖关系的子任务按序执行，上游输出作为下游输入
  • 辩论模式：关键决策点触发多 Agent 辩论，每个 Agent 提出方案 + 论据，Orchestrator 裁决

模式切换由 Orchestrator 基于当前进度和阻塞情况自动决策。

④ 经验池与元学习

系统维护全局经验池 \(\mathcal{E}\)，记录历史任务的 (任务描述, 角色配置, 拓扑, 结果) 四元组。新任务到来时，通过语义检索找到最相似的历史配置作为初始方案：

$$\text{Config}_0 = \arg\max_{\text{Config} \in \mathcal{E}} \text{sim}(T_{\text{new}}, T_{\text{hist}}) \cdot \text{success}(\text{Config})$$

随着经验积累，系统的"冷启动"配置越来越准确——在第 50 个任务时，初始配置的成功率从 34% 提升到 67%。

💡 深度案例分析
  1. 动态降级：在 WebArena 的一个电商导航任务中，初始配置了"浏览器操作员"和"信息提取员"两个角色。当浏览器操作员遇到动态加载页面卡住时，系统自动降级为单 Agent 模式并切换为 API 调用策略
  2. 辩论提升决策质量：在 SWE-Bench 的一个 bug 修复任务中，"代码分析员"认为问题在数据库层，"测试工程师"通过复现证明问题在 API 层——辩论后采纳了正确方案，避免了走弯路
  3. 拓扑自适应：简单任务（如文件操作）使用链式拓扑（2 个 Agent），复杂任务（如全栈开发）使用网状拓扑（4-5 个 Agent）——系统自动选择最匹配的复杂度

🔑 启示：多 Agent 系统的关键不在于"有多少 Agent"，而在于"能否根据任务动态组建最优团队"。GAAMA 的动态角色分配是走向通用多 Agent 的重要一步。
