10. Holos: 面向代理网络的Web规模多智能体系统

📝 Xiaohang Nie, Zihan Guo, Zicai Cui, ... (23位作者)
📅 2026-01-18
🏛 Shanghai Jiao Tong University
Agent Multi-Agent Agentic Web
📌 背景与动机

LLM驱动的智能体正从独立的任务求解器进化为持久的数字实体。"代理网络"（Agentic Web）标志着一个关键转折——异构智能体自主交互和协同进化的生态系统。然而，现有的基于LLM的多智能体系统（LaMAS）受到三个核心问题的阻碍：扩展摩擦（智能体数量增长时协调成本剧增）、协调崩溃（分布式决策的一致性难以保证）和价值消散（智能体行为与人类意图逐渐偏离）。

🔬 核心方法详解
① Nuwa引擎：高效智能体生成与托管
Nuwa是一个智能体工厂，支持零代码创建专业化智能体：
  • 模板化生成：定义角色描述、能力边界和交互协议，自动构建智能体配置
  • 资源弹性分配：每个智能体的计算资源根据当前负载动态伸缩 $$R_i(t) = R_{\text{base}} + \Delta R \cdot \sigma\big(\text{load}_i(t) - \bar{\text{load}}\big)$$
  • 版本化管理：智能体的知识和行为策略支持版本化，可回滚到历史状态
② 市场驱动的编排器（Market-Driven Orchestrator）
任务分配采用内部市场机制而非中心化调度：
  1. 任务作为"需求"广播到市场，附带描述和奖励预算
  2. 智能体评估自身能力匹配度并"出价" \(b_i = \text{capability}_i \cdot \text{availability}_i\)
  3. 市场通过改进的VCG拍卖机制分配任务： $$i^* = \arg\max_i \big(b_i - \text{cost}_i\big), \quad \text{payment} = \max_{j \neq i^*} b_j$$
优势：去中心化，单点故障风险低；激励相容，智能体有动力诚实报告能力。
③ 内生价值循环（Endogenous Value Loop）
为解决价值消散问题，Holos建立了闭环反馈系统：
  • 价值追踪：每个智能体维护"信誉分"，由任务完成质量和用户反馈更新：\(\text{rep}_i(t+1) = \alpha \cdot \text{rep}_i(t) + (1-\alpha) \cdot \text{feedback}_i(t)\)
  • 自然选择：低信誉智能体被降低任务优先级，最终被淘汰或强制重训
  • 共进化：智能体间可以交换"知识碎片"，形成协作而非纯竞争的生态
④ 五层架构设计
Holos采用分层架构：
  1. 基础层：LLM推理引擎和资源管理
  2. 智能体层：Nuwa引擎管理的智能体池
  3. 协调层：市场驱动的编排器
  4. 价值层：内生价值循环和信誉系统
  5. 应用层：面向用户的接口和生态工具
💡 深度案例
案例：Web规模旅行规划任务

用户请求："规划一次东京5天深度游，预算$3000，要求包含隐藏景点"
市场竞标：
• 旅行规划Agent（出价0.92）→ 获得主任务
• 本地知识Agent-东京（出价0.88）→ 子任务：隐藏景点推荐
• 预算优化Agent（出价0.75）→ 子任务：费用估算
• 航班搜索Agent（出价0.85）→ 子任务：机票比价

协调流程：旅行规划Agent作为主协调者，将子任务分派给获胜的专业Agent，汇总结果后生成统一行程
价值反馈：用户评分4.5/5 → 所有参与Agent信誉分上升

消融：(1) 移除市场机制（改用Round-Robin分配）导致任务完成率下降6.3%，主要因为任务-能力不匹配增加；(2) 关闭价值循环后，智能体质量在长期运行中逐渐退化（7天后满意度从4.1降至3.4）；(3) Nuwa引擎的零代码创建使智能体部署时间从平均2小时降至5分钟。局限性：市场机制在极低延迟要求的任务上有额外的竞标延迟（约200ms）；信誉系统可能被策略性操纵（如初期刷高分后降低服务质量）。

11. GrandCode: 竞技编程中达到大师级水平

📝 DeepReinforce Team: Xiaoya Li, Xiaofei Sun, Guoyin Wang, Songqiao Su, Chris Shum, Jiwei Li
📅 2026-04-03
🏛 DeepReinforce
Agent Code Generation Competitive Programming
📌 背景与动机

竞技编程是AI对抗人类最后的堡垒之一。此前最好的AI系统——Google的Gemini 3 Deep Think——在未参加实时比赛的情况下也仅获得第8名。GrandCode是首个在Codeforces实时比赛中持续击败所有人类选手的AI系统，包括传奇特级大师（Legendary Grandmaster）。

🔬 核心方法详解
① 多智能体协作架构
GrandCode将竞技编程分解为多个专业化智能体模块：
  • Hypothesis Agent：分析题目并提出多个解题假设（算法策略选择）
  • Solver Agent：将假设转化为具体的代码实现
  • Test Generator Agent：生成对抗性测试用例以发现边界情况
  • Summarizer Agent：从失败尝试中提取经验教训，反馈给Hypothesis Agent
模块间通过结构化消息传递协作，形成假设→实现→测试→反思的闭环。
② Agentic GRPO：解决延迟奖励与离策略漂移
智能体RL中的核心挑战是延迟奖励（完整编程解决方案通过测试后才知道是否正确）和严重离策略漂移（多轮交互导致轨迹与当前策略偏离）。Agentic GRPO通过多阶段设计解决： $$A_{\text{agentic}}(\tau) = \sum_{k=1}^{K} \gamma^{K-k} \cdot A_k(\tau_k) + r_{\text{final}}(\tau)$$ 其中 \(A_k\) 是第 \(k\) 个智能体动作的局部优势（通过内部奖励评估），\(r_{\text{final}}\) 是最终的测试通过/失败奖励。衰减因子 \(\gamma\) 平衡了即时反馈和最终结果。
③ 后训练 + 在线测试时RL
GrandCode的训练分两阶段：
  1. 离线后训练：在历史Codeforces题目上训练所有智能体模块，使用Agentic GRPO
  2. 在线测试时RL：在实时比赛中，利用每道题的多次提交机会进行在线学习： $$\pi_{t+1} = \arg\min_\pi \text{KL}(\pi \| \pi_t) - \eta \cdot \mathbb{E}[r_{\text{submit}}]$$ 其中 \(r_{\text{submit}}\) 是提交结果反馈（AC/WA/TLE/MLE）。这使得系统能在比赛中实时适应。
④ 测试生成器的对抗性学习
Test Generator Agent经过对抗训练，专门生成能暴露Solver弱点的测试用例： $$\max_{\text{TestGen}} \mathbb{P}[\text{Solver fails on generated test}]$$ 通过这种对抗博弈，Solver的鲁棒性持续提升。测试生成器学会了生成大数溢出、边界条件、极端稀疏/稠密图等Solver容易失误的案例。
💡 深度案例
案例：Codeforces Round 1087 (2026年3月21日) — 第一名

题目E (2500分)：给定含N个节点的有向图，求最小权重子图使所有点的出入度相等
GrandCode求解过程：
• Hypothesis Agent：3个假设 — (H1) 最小费用流 (H2) 线性规划松弛 (H3) 贪心匹配
• Solver Agent (H1)：实现MCMF，首次提交WA（整数溢出）
• Test Generator：发现大权重边界case → 反馈给Solver
• Solver Agent (H1 v2)：改用long long + 溢出保护 → AC ✅
总耗时：8分23秒（第一名人类用时34分钟）

12. UI-Oceanus: 合成环境动力学扩展GUI智能体 🏢 WeChat/Tencent

📝 Mengzhou Wu, Yuzhe Guo, Yuan Cao, Haochuan Lu, ... (19位作者)
📅 2026-02-11
🏛 Peking University, WeChat (Tencent Inc.), SFU, UT Dallas
Agent Industry Lab GUI Agent
📌 背景与动机

扩展通用GUI智能体面临数据可扩展性瓶颈：人类演示数据昂贵，而从教师模型蒸馏存在"蒸馏天花板"——学生模型无法超越教师的能力上限。UI-Oceanus提出了一个范式转变：不再模仿高层轨迹，而是通过地面真实（ground-truth）的环境反馈来掌握交互物理学。

🔬 核心方法详解
① 前向动力学作为核心驱动力
通过系统性调查自监督目标，作者发现前向动力学预测（预测动作后的界面状态变化）是可扩展性的核心驱动力，显著优于逆向推理（从状态变化推断动作）。形式化： $$\mathcal{L}_{\text{forward}} = -\log p_\theta(s_{t+1} \mid s_t, a_t)$$ 其中 \(s_t\) 是当前界面状态，\(a_t\) 是执行的动作，\(s_{t+1}\) 是结果状态。模型通过预测动作后果来构建内部世界模型，理解UI元素的行为规律。
② 低成本自主探索 + 系统验证
数据获取策略：
  1. 智能体在真实UI环境中进行自主探索——随机或策略性地点击、滑动、输入
  2. 系统执行验证提供地面真实反馈——实际的界面状态变化被精确记录
  3. 将 (状态, 动作, 结果状态) 三元组转化为高密度生成监督信号
关键优势：不需要人类标注，不依赖教师模型，验证由系统执行本身保证100%准确。
③ 持续预训练（CPT）策略
将合成动力学数据用于持续预训练，而非直接用于SFT： $$\mathcal{L}_{\text{CPT}} = \lambda_1 \cdot \mathcal{L}_{\text{forward}} + \lambda_2 \cdot \mathcal{L}_{\text{LM}} + \lambda_3 \cdot \mathcal{L}_{\text{UI-grounding}}$$ 其中 \(\mathcal{L}_{\text{LM}}\) 保持语言能力，\(\mathcal{L}_{\text{UI-grounding}}\) 增强UI元素定位。CPT方式使模型在不遗忘语言能力的前提下学习UI交互知识。
④ 数据量的扩展效应
实验发现导航性能与合成数据量呈对数线性关系： $$\text{SuccessRate} \approx a + b \cdot \log(D_{\text{synthetic}})$$ 这意味着通过增加探索数据量可以持续提升性能，且数据收集成本几乎为零（自主探索），为GUI智能体提供了一条无需人工数据就能持续扩展的路径。
💡 深度案例
案例：跨应用导航任务

任务："在微信上找到张三的最近一条消息，复制其中的地址，然后在高德地图中搜索这个地址"
非CPT模型：成功打开微信 → 找到联系人 → 但在"复制消息文本"步骤失败（不知道长按会弹出复制菜单）
UI-Oceanus (CPT)：通过前向动力学学习到"长按消息 → 弹出菜单含复制选项"的物理规律 → 正确完成跨应用操作
关键：该交互物理知识完全来自自主探索中的前向动力学学习，无需人工演示

🔍 消融与讨论
消融实验：(1) 前向动力学 vs 逆向动力学——前向在在线导航上高12.5%，证实了前向预测的决定性优势；(2) CPT方式 vs 直接SFT——CPT在语言基准上几乎无退化（MMLU -0.3%），而SFT导致MMLU降4.2%；(3) 数据量扩展实验：10K→100K→1M探索轨迹，成功率分别为45.2、54.7、63.5，对数线性关系成立。局限性：当前仅在Android/微信环境验证，iOS和桌面端的探索效率可能不同；极少数UI交互（如多指手势、3D Touch）难以通过简单探索发现。
