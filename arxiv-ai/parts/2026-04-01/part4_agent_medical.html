Marco DeepResearch: Unlocking Efficient Deep Research Agents via Verification-Centric Design

👤 Bin Zhu, Qianghuai Jia, Tian Lan, Junyang Ren, Feng Gu, Feihu Jiang, Longyue Wang, Zhao Xu, Weihua Luo 📅 2026-03-30 · arXiv: 2603.28376 🏛 Alibaba Group AgentDeep ResearchVerification

📌 背景与动机

深度研究（Deep Research）Agent 旨在自主完成复杂信息搜索和报告撰写，但现有系统面临两大瓶颈：(1) 生成不忠实——Agent 检索到的信息与最终生成的报告之间存在大量幻觉和不一致；(2) 效率低下——为保证质量，需要大量冗余检索和多轮验证，计算成本极高。Marco DeepResearch 提出以验证为中心的设计理念，在数据、训练和推理三个层面引入系统性验证机制。

🔬 核心方法详解

① 数据层验证（Data-Level Verification）
构建"可验证搜索轨迹"训练数据。对于每个查询-报告对，回溯验证报告中每个声明的来源： $$\text{VerifyScore}(c, D) = \max_{d \in D} \text{NLI}(d, c)$$ 其中 \(c\) 是报告中的声明，\(D\) 是检索到的文档集。只保留验证分数超过阈值的声明。通过这一过程，构建了约 50K 条高质量"检索-验证-生成"三元组训练数据，确保训练信号本身就是忠实的。
② 训练层验证（Training-Level Verification）
在 SFT 之后引入验证感知的 RL 阶段。奖励函数融合三个维度： $$R = \alpha R_{\text{faithfulness}} + \beta R_{\text{relevance}} + \gamma R_{\text{completeness}}$$ 其中忠实度奖励通过对比生成内容与检索文档计算；相关性奖励由查询与报告的语义相似度给出；完整性奖励评估关键信息的覆盖率。关键创新：在 RL rollout 中，Agent 每次生成声明后立即进行自验证（self-verification）步骤 — 检查该声明是否有检索证据支持，如果没有则触发补充检索或删除该声明。
③ 推理层验证（Inference-Level Verification）
部署时采用"生成-验证-修订"循环：
  1. 初始生成: Agent 基于检索结果生成初版报告
  2. 逐句验证: 验证器检查每个声明的证据链，标记无支持声明
  3. 定向修订: 仅对标记部分进行补充检索和重写，避免全文重新生成
这一设计将测试时计算（test-time compute）精准投入到最需要的地方，而非均匀分配。实验表明，相比"全文重生成 3 次"策略，定向修订仅用 40% 的计算量即达到相同质量。

💡 深度案例

  • 查询: "比较 2025 年中国三大电池厂商（宁德时代、比亚迪、中航锂电）的固态电池技术路线和量产时间表"
  • 无验证 Agent: 检索 12 篇网页，生成 2000 字报告，但 4 处数据引用错误（如混淆宁德时代的半固态 vs 全固态路线），忠实度 61%
  • Marco DeepResearch: 检索 8 篇网页 → 初版生成 → 验证发现 2 处无支持声明 → 补充检索 3 篇 → 修订最终报告，忠实度 94%
  • 计算开销: 仅比无验证版多 35% 的 token 消耗

AgentSwing: Adaptive Parallel Context Management Routing for Long-Horizon Web Agents

👤 Zhaopeng Feng, Liangcai Su, Zhen Zhang, et al. 📅 2026-03-29 · arXiv: 2603.27490 🏛 Alibaba DAMO Academy, NUS AgentWeb AgentContext Management

📌 背景与动机

随着 LLM 演变为自主 Web Agent，长时序信息搜索中的有限上下文窗口管理成为关键瓶颈。现有方法通常承诺使用单一固定策略（如只保留最近 N 步、或只保留摘要），但信息搜索过程中上下文的有用性是动态变化的——有时需要详细原文，有时只需摘要，有时需要回溯到早期步骤。静态策略无法适应这种变化。

🔬 核心方法详解

① 状态感知触发器（State-Aware Trigger）
设计触发条件检测何时需要进行上下文管理决策。监测三个信号： $$\text{Trigger}(t) = \mathbb{1}[\text{CtxLen}(t) > \tau_1] \lor \mathbb{1}[\Delta_{\text{info}}(t) > \tau_2] \lor \mathbb{1}[\text{StuckScore}(t) > \tau_3]$$ 其中 \(\text{CtxLen}\) 是当前上下文长度，\(\Delta_{\text{info}}\) 衡量新获取信息的信息量变化，\(\text{StuckScore}\) 检测 Agent 是否陷入循环。只在触发时进行上下文管理，非触发时保持原始上下文不变。
② 并行分支扩展（Parallel Branch Expansion）
在每个触发点，并行生成多个上下文管理候选分支：
  • 全保留分支: 不做任何压缩，保留完整上下文
  • 摘要分支: 将历史步骤压缩为摘要
  • 选择性保留分支: 仅保留与当前子目标最相关的历史步骤
  • 重置分支: 丢弃所有历史，从当前状态重新开始
每个分支独立执行接下来 \(k\) 步（前瞻 lookahead），产生不同的执行轨迹。形式化地，在触发点 \(t\)，生成候选集 \(\{c_1, \ldots, c_m\}\)，每个候选执行 \(k\) 步得到前瞻轨迹 \(\tau_i^{(t:t+k)}\)。
③ 前瞻路由选择（Lookahead Routing）
基于前瞻轨迹的质量选择最佳分支继续执行： $$c^* = \arg\max_{c_i} \text{Score}(\tau_i^{(t:t+k)})$$ 评分函数综合考虑：任务进展（是否更接近目标）、信息增益（获取了多少新信息）和效率（步数消耗）。选择后，丢弃其他分支，仅保留最佳分支的上下文继续。设计直觉：这是一种"上下文管理的 beam search"，用少量额外计算（并行前瞻）换取更好的长期决策。

  • 固定摘要策略: 第 8 步摘要后丢失了 Reddit 帖子的具体 URL → 第 12 步需要回访时找不到 → 总共 23 步完成
  • AgentSwing: 第 8 步触发 → 并行尝试摘要/选择性保留/全保留 → 前瞻发现选择性保留（保留 URL 和关键引用）最优 → 第 15 步完成（减少 35% 交互轮数）
  • 关键优势: 自适应选择避免了"摘要过度丢失细节"和"全保留超出上下文"两个极端

GraphWalker: Agentic Knowledge Graph Question Answering via Synthetic Trajectory Curriculum

👤 Shuwen Xu, Yao Xu, Jiaxiang Liu, Chenhao Yuan, Wenshuo Peng, Jun Zhao, Kang Liu 📅 2026-03-30 · arXiv: 2603.28533 🏛 Chinese Academy of Sciences (ICT/CAS) AgentKnowledge GraphQA

📌 背景与动机

知识图谱问答（KGQA）中的 Agent 方法要求模型迭代地与知识图谱交互：选择实体、遍历关系、收集证据、推理答案。但现有方法面临训练数据稀缺和推理泛化两大挑战。基于提示的方法（prompting）缺乏自我纠错能力，基于微调的方法受限于有限的人工标注轨迹。

🔬 核心方法详解

① 合成轨迹生成（Synthetic Trajectory Generation）
从知识图谱中自动生成大量 Agent 交互轨迹作为训练数据。方法：
  1. 从目标实体出发，在 KG 上反向随机游走生成路径 \(p = (e_k, r_k, \ldots, r_1, e_0)\)
  2. 使用 LLM 将路径转化为自然语言问题 \(q\)
  3. 模拟 Agent 的正向遍历过程：从起始实体出发，按步选择关系和实体，记录完整交互轨迹
  4. 在轨迹中注入失败分支（错误选择 → 发现错误 → 回溯 → 正确选择），训练自纠错能力
形式化地，生成轨迹 \(\tau = [(a_1, o_1), \ldots, (a_T, o_T)]\)，其中 \(a_t\) 是 Agent 动作，\(o_t\) 是 KG 返回的观察。
② 课程学习调度（Curriculum Learning Schedule）
按难度递增安排合成轨迹的训练顺序： $$\text{Difficulty}(\tau) = \alpha \cdot |p| + \beta \cdot |\text{backtrack}(\tau)| + \gamma \cdot \text{deg}(e_0)$$ 其中 \(|p|\) 是路径长度（跳数），\(|\text{backtrack}|\) 是回溯次数，\(\text{deg}(e_0)\) 是起始实体的度数（度越大 → 选择空间越大 → 越难）。训练分三阶段：
  • Phase 1: 1-2 跳、无回溯的简单轨迹
  • Phase 2: 2-3 跳、有 1 次回溯的中等轨迹
  • Phase 3: 3-5 跳、多次回溯的困难轨迹
③ KG 感知动作空间约束（KG-Aware Action Space）
在推理时，Agent 的动作空间被约束为当前实体的合法邻居关系： $$\mathcal{A}(e_t) = \{(r, e') \mid (e_t, r, e') \in \mathcal{KG}\} \cup \{\text{BACKTRACK}, \text{ANSWER}\}$$ 这避免了 Agent 生成不存在的实体或关系（幻觉），同时保留了回溯和提交答案的选项。在生成每一步时，将合法动作列表注入到 prompt 中，模型通过受限生成（constrained decoding）选择动作。

💡 深度案例

  • 问题: "哪部电影的导演也导演了《泰坦尼克号》？" (2-hop)
  • 轨迹: 搜索"泰坦尼克号" → 找到实体 → 选择"director"关系 → James Cameron → 选择"directed"关系 → 得到所有电影列表
  • 回溯示例: 初次误选"producer"关系 → 观察结果不匹配 → BACKTRACK → 改选"director" → 成功
  • Prompting baseline: 在高度数实体处困惑（James Cameron 有 30+ 关系），选择正确率仅 52%
  • GraphWalker: 经课程训练后，高度数实体处正确率 81%

Towards a Medical AI Scientist

👤 Hongtao Wu, Boyun Zheng, Dingjie Song, Yu Jiang, Jianfeng Gao, Lei Xing, Lichao Sun, Yixuan Yuan 📅 2026-03-30 · arXiv: 2603.28589 🏛 CUHK, Microsoft Research, Stanford University MedicalAgentAI Scientist

📌 背景与动机

AI Scientist——能自主生成假设、设计实验、分析结果并撰写论文的系统——是 AI 研究的圣杯之一。但现有的 AI Scientist 系统（如 Sakana AI 的 The AI Scientist）是领域无关的（domain-agnostic），在临床医学中面临特殊挑战：伦理约束（不能随意实验人体）、数据隐私（EHR 数据不可自由访问）、因果推理需求（相关性不等于因果性在医学中尤为关键）。本文提出首个面向医学领域的 AI Scientist 框架设计。

🔬 核心方法详解

① 医学知识增强假设生成（Knowledge-Augmented Hypothesis Generation）
假设生成模块整合三类知识源：
  • 文献知识: 通过 PubMed RAG 获取最新研究进展
  • 临床知识图谱: UMLS、DrugBank 等结构化知识
  • 临床指南: UpToDate、WHO guidelines 等规范性知识
假设生成遵循"PICO 框架"（Patient, Intervention, Comparison, Outcome）： $$H = \text{LLM}(\text{RAG}(\text{PubMed}, q) \oplus \text{KG}(\text{UMLS}, e) \oplus \text{Guide}(\text{UpToDate}, t))$$ 每个生成的假设附带可检验性评分和伦理风险评估。
② 回顾性实验设计器（Retrospective Experiment Designer）
由于医学中前瞻性实验受伦理限制，系统专注于回顾性研究设计。给定假设，自动设计：
  • 纳入/排除标准（用 ICD 代码表示）
  • 混杂因素列表和调整策略（倾向性评分匹配、IV 回归等）
  • 统计分析计划（主要终点、次要终点、效应量估计）
$$\text{Design} = (S_{\text{inclusion}}, S_{\text{exclusion}}, \mathcal{C}_{\text{confounders}}, \mathcal{T}_{\text{analysis}})$$ 设计直觉：将临床研究方法学的知识编码到 Agent 的规划过程中，确保生成的研究方案在方法学上是合理的。
③ 因果推断引擎（Causal Inference Engine）
在数据分析阶段引入因果推断框架。系统自动构建因果 DAG（有向无环图）并选择合适的因果估计方法： $$\text{ATE} = \mathbb{E}[Y(1) - Y(0)] \approx \frac{1}{n}\sum_{i=1}^n \frac{T_i Y_i}{e(X_i)} - \frac{(1-T_i)Y_i}{1-e(X_i)}$$ 其中 \(e(X_i)\) 是倾向性得分（propensity score）。系统还实施多重敏感性分析（E-value、负控制结局等）来评估因果结论的稳健性。关键约束：当因果推断无法满足不可混淆性假设时，系统主动标注结论为"关联性"而非"因果性"。

💡 深度案例

📝 MIMIC-IV 上的自主研究示例
  • 假设生成: "ICU 患者中，早期使用低剂量皮质类固醇（入院 24h 内）与脓毒症相关死亡率降低相关"
  • 实验设计: 纳入 ICD-10 A41.x 脓毒症患者，排除 <18 岁和既往类固醇使用；倾向性评分匹配 1:1
  • 结果: 匹配后 n=3,284 对，ATE = -0.038 (95% CI [-0.062, -0.014], p=0.002)，死亡率降低 3.8%
  • 与人类研究对比: 结论方向与 ADRENAL 试验（2018, NEJM）一致，效应量相近
  • 论文生成: 自动撰写符合 STROBE 规范的观察性研究论文

EpiScreen: Early Epilepsy Detection from Electronic Health Records with Large Language Models

👤 Shuang Zhou, Kai Yu, Zaifu Zhan, Huixue Zhou, Min Zeng, Feng Xie, Zhiyi Sha, Rui Zhang 📅 2026-03-30 · arXiv: 2603.28698 🏛 University of Minnesota MedicalEHREpilepsy

📌 背景与动机

癫痫（Epilepsy）与心因性非癫痫发作（PNES）在临床表现上极为相似——都表现为发作性事件，但需要完全不同的治疗策略。误诊极为常见，平均诊断延迟达 7-10 年，导致不必要的抗癫痫药物使用和患者致残。虽然 AI-ECG 已被用于心脏疾病筛查，但基于电子健康记录（EHR）的癫痫早期识别仍是空白——这正是 LLM 可以发挥作用的地方。

🔬 核心方法详解

① EHR 纵向特征提取（Longitudinal EHR Feature Extraction）
将患者的 EHR 时间线转化为结构化特征向量。从三个数据源提取：
  • 临床笔记: 使用 Med-PaLM 2 提取症状描述中的 23 个癫痫相关特征（如发作频率、先兆症状、意识状态等）
  • 诊断代码: ICD-10 G40.x（癫痫）、F44.5（PNES）及相关合并症
  • 用药记录: 抗癫痫药物（ASMs）使用模式、药物切换频率
时间线表示为 \(\mathbf{X} = [\mathbf{x}_{t_1}, \ldots, \mathbf{x}_{t_n}]\)，每个时间戳包含上述特征的向量化表示。
② LLM 驱动的临床推理（LLM-Driven Clinical Reasoning）
将提取的特征序列转化为临床叙事文本，输入 LLM 进行多步推理： $$P(\text{Epilepsy}|\mathbf{X}) = \text{LLM}(\text{Narrative}(\mathbf{X}) \oplus \text{ClinicalGuideline})$$ 关键设计：引入"诊断标准对照"（Diagnostic Criteria Alignment）prompt，将 ILAE 2017 分类标准嵌入到推理过程中，要求模型逐条对照每个诊断标准并给出支持/反对证据。这模拟了神经科医生的鉴别诊断思维过程。
③ 可解释预测框架（Interpretable Prediction Framework）
每个预测伴随结构化解释： $$\text{Output} = (\hat{y}, \text{Confidence}, \text{Evidence}, \text{Uncertainty})$$ 其中 Evidence 包含支持诊断的关键 EHR 记录引用，Uncertainty 量化为模型在多次推理中的一致性（5 次采样的 self-consistency score）。当一致性低于阈值时，系统标注为"需要进一步检查"而非给出确定诊断。

  • 最终确认: 视频 EEG 确认 PNES，提前 2.3 年做出正确鉴别
