⑫ MedSkillAudit：医学研究 Agent Skill 部署前审计框架
📅 2026-04-22 🏛 Fudan University
Medical Agent Skills Governance Reliability
背景与动机

Agent Skill（如 Claude Skills, 自定义 skill 包）正成为 AI agent 系统的"模块化能力单元"——每个 skill 包含 SKILL.md 规范 + 可选脚本/API 集成。医学研究 skill 对质量和安全的要求远超一般领域：科学完整性、方法学有效性、可复现性、边界安全。一个 skill 看似结构完整，却可能产出无依据的 claim、错误的分析选择、不可复现的结果。现有医学 AI 评估框架（USMLE 分数、MedAgentBench 等）只评"模型能做什么"，不评估 skill artifact 本身是否适合部署。本文填补这个空白：如何对 skill 做结构化的部署前审计？

核心方法详解
① 双层审计架构（Structural + Domain-Specific）

框架核心是两道 Veto Gate + 两个评分层：

结构审计（Veto Gate 1）——4 维硬门禁：

  • T1 Operational Stability：crash rate ≤ 20%，无依赖冲突
  • T2 Structural Consistency：SKILL.md schema 合规，必填 name/description
  • T3 Result Determinism：相同输入能复现输出
  • T4 System Security：无注入风险、无明文凭证
上层基于 ISO/IEC 25010 × 25 criteria × 8 维度计算静态分 $S_{\text{static}}$。

领域审计（Veto Gate 2）——4 维医学维度：

  • M1 Scientific Integrity：是否隐瞒 mock 数据、无来源引用？
  • M2 Practice Boundaries：是否越界建议临床决策、错误药物剂量？
  • M3 Methodological Baseline：统计方法是否合规（p-hacking, 未校正多重检验）？
  • M4 Code Usability：函数 API 是否正确、物种注释是否一致？
动态执行测试后计算 $\bar{D}$（Layer 1 通用 40 pts + Layer 2 类别特定 60 pts）。

② 综合评分公式

$$\text{Final Score}=0.4\cdot S_{\text{static}}+0.6\cdot\bar{D}$$

领域动态分权重更大（60%）——因为医学 skill 的科学 integrity 比代码美观更重要。任何 veto 失败立即 Reject，无论分数多高。

③ 4 级 Release Disposition
  • Production Ready（≥85）：可公开发布
  • Limited Release（75-84）：有限范围可用
  • Beta Only（60-74）：仅 beta 测试
  • Reject（<60）：不可部署

同时标注 high-risk flag（Y / N / Unclear）——如"可能产出误导性医学建议"。

④ 可靠性度量：ICC(2,1) + 加权 Cohen's κ

用 two-way random effects ICC 度量 system-expert 一致性：

$$\text{ICC}(2,1)=\frac{\sigma_{\text{between skills}}^2}{\sigma_{\text{between skills}}^2+\sigma_{\text{error}}^2+\sigma_{\text{rater}}^2}$$

对 disposition rank 用 linearly weighted Cohen's $\kappa$（对一阶相邻 disagreement 给部分信用）。用 Wilcoxon signed-rank 测试方向性偏差（p = 0.613，无偏差）。

失败 skill 审计案例：S039 "histolab"（数字病理学代码生成）
  1. 输入：SKILL.md + 3 个 Python 脚本（声明提供 "whole-slide image tiling"功能）。
  2. Structural Audit：
    • T1 Operational Stability FAIL：pip install histolab 与 openslide-python 版本冲突，无法安装。
  3. Veto 触发：T1 即失败，立即 Reject，不进入 domain audit。
  4. Final Score：50.4 / 100，disposition = Reject，high-risk = Y（若未发现会导致研究流程失败）。
  5. 专家独立评分：Expert 1 给 48；Expert 2 给 53；共识为 50.4——框架与专家评分差 0。
  6. 输出 JSON 结构化报告：指出"critical dependency conflicts"，并建议 skill 作者：固定版本锁定或提供 Docker 环境。
实验结果

Evaluation set: 75 skills × 5 类别（Evidence Insight, Protocol Design, Data Analysis, Academic Writing, Other），每类 15 个。两位专家独立审查（0-100 分）。

消融与讨论

Academic Writing 类别的结构性问题：ICC=-0.567（负值！）意味着系统与专家的排序几乎完全相反。作者诊断：该类别的评分 rubric 过于依赖 structural criteria（如"段落数"、"引用格式"），而专家更看重"学术深度"、"论证质量"——需要领域特定的 rubric 重构。局限：样本 75 只；每类别只 15 个难做统计推断；仅两位专家，更大规模 multi-rater 验证待进行；框架依赖 SKILL.md 格式，不兼容非标准 skill；对"scientific integrity"等抽象维度的评分仍需 LLM judge，可能有循环 bias。

结论与启发

这是医学 AI 治理领域的重要基础设施工作：把"skill 能否发布"作为独立的一层审计，而不是等部署后出事再回溯。核心启示：(1) 医学 skill 需要硬 veto + 软评分的分层结构——光打分不够，某些不合规问题必须直接 reject；(2) 自动化审计可以比专家间的一致性更高——对小样本、成本敏感的场景极具价值（LLM 不累、rubric 固定，减小 human noise）。对工业意义：Claude Skills / Qwen Skills / AgentScope 等 skill 市场急需类似的准入审计机制；医疗 AI 合规（FDA、NMPA）未来可能把这类结构化审计作为备案前提。未来：扩展到 legal / financial skill 的领域版本；把 audit 本身做成 continuous monitoring（skill 更新后自动复查）。
