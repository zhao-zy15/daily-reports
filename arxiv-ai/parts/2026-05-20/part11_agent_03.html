① 背景与动机

当前 LLM agent 的 "skill" 大多是 Markdown 文件（Anthropic Skill、OpenAI Operator skill、Cursor rules）：自然语言写一段操作流程，依赖模型自己解读 + 自我约束。但实际部署遇到三大问题：(i) 约束不可执行——"修改前请备份"这种规则模型可能跳过；(ii) 状态不显式——agent 不知道自己在哪个 phase，无法 recover；(iii) 工具暴露过宽——所有工具一直可见，导致 token 浪费且诱发误用。本文提出 Formal Skill：把 Markdown skill 升级为带 schema、executable hooks、phase state machine 的可编程运行时插件。

② 核心方法详解

① Action Plane（动作平面）

每个 Formal Skill 暴露typed action schema（JSON Schema）+ Python executor。模型只看到 JSON schema（"collect evidence", "apply unified patch"），Python 端做实际执行 + 路径检查 + 命令白名单 + patch-shape validation。模型无法绕过执行端的检查。

② Hook-Governed Phase State Machine（CODEREPAIROPS 示例）

Formal Skill 维护 skill-local runtime state（phase, verification status, produced artifacts, gate-failure reasons）。before_llm_call hook 按 phase 过滤可见工具：

PhaseVisible actionsHook policy
reproducecollect evidence仅 evidence 可见。命令或日志证据更新 failure signature → 进入 patch
diagnosecollect, patchrouted optional 阶段。通用 mutation 工具隐藏
patchcollect, patch必须用 unified patch 工具；guards 拒绝保护路径 / 缺失 hunk / 过大变更。成功 → verify
verifyverify仅 verification check 可见。通过 → report；失败 → 回 patch
reportwrite artifacts, doneartifact 写入与 completion 可见。done 被阻塞直到 verification 通过

这样约束变成 action semantics 而非可被忽略的 prompt。

③ Skill-Local Runtime State（recovery 友好）

每个 skill 维护独立 runtime state（不污染 main agent context）。当 verification 失败，runtime 知道精确 failure reason，可决策：继续 collect evidence、回 patch、enter reporting、block finalization。recovery 不再依赖模型自己 reason "我现在该干什么"。

④ Token Efficiency Mechanism

三个 token 节省来源：(1) phase-specific tool visibility（每 LLM call 只注入相关 tools）；(2) compact phase guidance（不是整段 skill.md，而是当前 phase 的 condensed instruction）；(3) narrower tool exposure（避免无关工具列表 inflation）。这些机制不依赖 transcript 压缩，是结构性的 token 节省。

③ 深度案例

Harness-Bench code-debugging 任务：
  1. Workspace：buggy_code.py 含 5 层叠加 bug，每轮只暴露当前层，agent 必须修复、validate，过了当前层才能 stop。
  2. FairyClaw + Formal Skill 行为：
    • Phase reproduce：仅 evidence 工具可见，agent 跑测试拿到 bug 1 的 stack trace；
    • Phase patch：仅 unified patch 工具，agent 写一个 minimal diff，guard 拒掉了"覆盖整个文件"的 sweep edit；
    • Phase verify：跑 test，bug 1 pass；verification_passed=true；
    • Phase report：写 artifact，completion gate 检查通过，stop。
  3. 对比 OpenClaw（无 phase state）：agent 一次性看到所有工具，倾向用 "edit_file" 把整个文件重写——失败率 30%+，且 token 多出 2-3 倍。
  4. 结果：FairyClaw + gpt-5.4 combined score 0.865 on debugging task；其他 agent 多在 0.5-0.7 之间。

⑤ 消融与讨论

Compression 已禁用：FairyClaw 的 token 节省不来自激进的 transcript compression，而是 routed Formal Skill + phase guidance + narrower tool exposure 的结构性收益。Skill authoring cost：Formal Skill 比 Markdown skill 编写门槛更高，需要写 JSON schema + Python executor + hook code。论文提出的 mitigation：用 LLM 把 Skill.md 自动编译成 Formal Skill 组件（schema + executor skeleton），让 user 仍然能从 human-readable Skill 开始。局限：仅在 75 task Harness-Bench 上评测，覆盖度有限；对开放式探索任务（无明确 phase 划分）适用性未验证；require 团队投入工程力构造 typed action interface。

⑥ 结论与启发

Formal Skill 把当前 agent ecosystem 中流行的 "prompt-only skill" 范式提升到 "executable, typed, state-machine 化的运行时插件"。对于工业 agent 平台（Anthropic Claude Skills、Cursor、Devin、Operator），这是下一步发展的明确方向：

  • 用 hooks 把"安全约束"做成 unbypassable 的 execution gate，不依赖模型自律
  • 用 phase state 把工具暴露动态收缩，显著节省 token
  • 用 skill-local runtime state 让 recovery 不依赖模型 reasoning

结合 ReBel（Agent-1，让 agent 显式输出 belief 状态）+ TriMem（Agent-2，多粒度 memory）+ Formal Skill（执行约束运行时），我们看到 2026 年 5 月 LLM agent 工程范式的明确演进方向：从"全靠模型自律"走向"约束硬编码 + 状态显式 + 工具受限"。这是工业可靠性的必经之路。
