① 背景与动机

Prefix cache 是现代 LLM serving 的关键优化——把不同请求间共享的 prompt 前缀 KV 状态复用，跳过 prefill 计算。但 GPU 显存稀缺，必须有淘汰策略。生产中普遍使用的 LRU 等策略把所有 KV 块视为同质：只看访问时间。本文作者用真实流量数据（ShareGPT + AgentBank）测得一个被忽视的事实——不同 token 类型的复用率差距高达 42 倍：system prompt 复用率 92.3%，user prompt 30.8%，response 27.8%，tool output 23.0%，而 chain-of-thought 推理痕迹仅 2.2%。把一个 system prompt 块和一个 CoT 块按相同优先级保留，等于浪费宝贵的 HBM。此外，多轮会话的复用几乎 100% 是 session-local，inter-turn 时间间隔服从 session 特定的 log-normal 分布（chat: P50=63s, P99=633s；agentic: P50=6s, P99=79s）——固定参数完全无法兼顾。

② 核心方法详解

① 多队列分区架构（Multi-Queue Routing）

SAECache 用基于规则的 partition 把每个 KV 块路由到三类专用队列：单轮模板队列（system prompt + tool def）、多轮历史队列（user/response/tool output）、CoT 推理队列。每个队列维护独立的优先级度量：

$$\text{Priority}(b) = w_{\text{type}}(t_b) \cdot w_{\text{queue}}(q_b) \cdot f(\Delta t_b, \tau_q)$$

其中 $w_{\text{type}}$ 是 token-type 权重（系统 prompt > user > response > CoT），$w_{\text{queue}}$ 是队列级权重，$f(\Delta t_b, \tau_q)$ 是与 log-normal session timescale $\tau_q$ 匹配的时间衰减函数。

② 语义感知的 token 加权（Online Reuse Weighting）

不依赖人工调参，作者把权重 $w_{\text{type}}(t)$ 设为在线学习参数。每当一个块被命中时，给该类型的权重一个正梯度；每当一个块被淘汰却随后被请求未中，给一个负梯度。形式上：

$$w_{\text{type}}^{(k+1)}(t) = w_{\text{type}}^{(k)}(t) + \eta \cdot [\mathbb{1}_{\text{hit}}(b) - \lambda \cdot \mathbb{1}_{\text{miss-after-evict}}(b)]$$

其中 $\eta$ 是学习率，$\lambda$ 是惩罚因子。整个机制不需要 offline profiling。

③ Tail-First eviction（结构性保护）

实证发现：相同 prefix 模板下，淘汰 prefix 头部块比淘汰尾部块更致命——因为头部一旦被驱逐，整段共享 prefix 就重新需要从头 prefill。SAECache 在每个队列内部强制 tail-first eviction，即同等优先级下优先驱逐尾部块。

④ Inter-Turn 自适应（log-normal 时间窗）

对每个 session 类型，SAECache 拟合 log-normal 分布 $\log \Delta t \sim \mathcal{N}(\mu_q, \sigma_q^2)$，并把生存期 $\tau_q$ 设为 $\exp(\mu_q + k\sigma_q)$（论文用 $k=2$，覆盖到 P95）。这样 agentic 工作流（短间隔）的块快速淘汰为后续高频请求让位，而 chat 工作流的块得到更长保留。

③ 深度案例

场景：一个混合 chat + agentic 的 LLM 服务，GPU KV 池仅能容纳 50% 的活跃前缀。
  • 请求 A（chat，刚结束一轮，下一轮预计 60s 后）：含 1000 token system prompt + 500 token 历史对话 + 200 token 当前 CoT 推理
  • 请求 B（agentic，每 8s 来一轮）：含 800 token system + tool def + 300 token tool output
LRU 行为：按时间排序，把刚生成完的 A 的 CoT 块也放到队尾"高优先级"；当 B 的 tool def 块被淘汰时，下一次 B 必须重 prefill 1100 token。
  1. SAECache 路由：A 的 system → 单轮模板队列（保留权重高）；A 的 CoT → 推理队列（权重 0.02 倍，立即可弃）；B 的 system + tool def → 单轮模板队列。
  2. 权重更新：online 学习 200 个请求后，CoT 权重 → 0.022，system 权重 → 0.923，差距 42×。
  3. 结果：A 的 CoT 块第一个被淘汰，B 的 tool def 永久保留，B 命中率从 LRU 的 41% 提升到 78%；端到端 TTFT 从 850ms 降到 360ms（2.36× 加速）。

⑤ 消融与讨论

消融实验显示三个组件的独立贡献：去掉 token-type weighting 损失 23% 增益；去掉自适应 queue weighting 损失 39%；去掉 tail-first 损失 11%。三者均不可或缺。局限：在线学习需要 ~500 个请求的 warm-up，冷启动期收益较小；CoT 复用率统计基于公开数据集 ShareGPT，对于代码补全等"高 CoT 复用"场景未必准确。

⑥ 结论与启发

SAECache 的核心洞见——"语义结构 + 时间分布 + 在线学习" 三轴联合驱动 KV cache 淘汰——为 LLM serving 系统打开了新的优化维度。它呼应了 LLM-2 Spherical KV 的理念：把 KV 看作 rate–distortion 分配问题；但走的是更朴素的工程路线：不需要改 attention kernel，只需在 serving runtime 加 200 行代码。对于阿里、字节、火山等推理平台，这是近期可落地的最高 ROI 改进点之一，预计将在下一波 vLLM/SGLang/TensorRT-LLM 优化中被吸收。
