🧠 纯文本大语言模型

1. Ring-Attention v4: Towards Infinite Context via Distributed KV-Cache Optimization

[arXiv PDF] [GitHub Code]
1. 背景与底层痛点 (Background & Motivation)

痛点分析： 随着 LLM 上下文窗口扩展到 1M+ token (如 Gemini 1.5)，传统的 FlashAttention 虽然解决了单卡显存爆炸的问题，但在分布式训练中仍然面临巨大的通信瓶颈。现有的序列并行 (Sequence Parallelism) 方法在切分 KV Cache 时，会导致 GPU 之间频繁的 P2P 通信，通信量随着 Context Length 的增长呈线性甚至超线性增长。

底层数学缺陷： 在标准的 Ring Attention 实现中，计算 Attention Score $QK^T$ 时，每个 device 需要在环上传递完整的 Key/Value 块。当序列长度 $L \to \infty$ 时，通信时间 $T_{comm} \approx O(L)$，而计算时间 $T_{comp} \approx O(L^2)$。虽然计算掩盖了通信，但在极大 Batch Size 下，跨节点带宽（Inter-node bandwidth）成为绝对瓶颈，导致 MFU (Model FLOPS Utilization) 骤降至 30% 以下。

2. 核心贡献 (Core Contributions)
  • Blockwise Overlapped Communication： 提出了一种新的异步流水线，将 KV 块的传输与 Attention Score 的计算完全解耦，实现了 98% 的通信-计算重叠率。
  • Sparse-Ring 拓扑： 针对因果掩码 (Causal Mask) 的稀疏性，动态跳过不需要注意力的环状节点，将通信复杂度从 $O(N)$ 降低到 $O(\log N)$。
  • 工程突破： 在 512 张 H100 上成功训练了 10M context window 的 7B 模型，MFU 保持在 55% 以上。
3. 方法详解 (Methodology - Hardcore)

核心架构：异步环状注意力

假设我们有 $P$ 个 GPU，序列长度为 $L$。每个 GPU 持有 $L/P$ 的 Query, Key, Value。 在第 $step$ 步，GPU $i$ 接收来自 GPU $i-1$ 的 Key 块 $K_{i-1}$。 标准公式：

$$ O_i = \text{Softmax}(\frac{Q_i K_{local}^T}{\sqrt{d}})V_{local} + \sum_{j \neq i} \text{Softmax}(\frac{Q_i K_j^T}{\sqrt{d}})V_j $$

本文引入了预测性预取 (Predictive Prefetching)。由于 Causal Mask 的存在，只有 $j \leq i$ 的块是有效的。 定义通信算子 $\mathcal{T}_{send}(K, i \to i+1)$ 和计算算子 $\mathcal{F}_{attn}(Q, K)$。 新的调度策略如下：

# Pseudo-code for Asynchronous Ring Attention req = dist.irecv(next_K, src=prev_rank) # 预取下一块 local_score = Q @ current_K.T # 计算当前块 (掩盖通信) dist.wait(req) # 等待传输完成 current_K = next_K # 指针轮转

此外，针对 RoPE (Rotary Positional Embedding)，文章提出在本地计算 attention 前再动态施加旋转矩阵，避免了在网络上传输经过旋转的高维数据，节省了显存带宽。

2. DPO vs KTO vs PPO: A Large-Scale Empirical Study on Reward Sparsity

[arXiv PDF] [GitHub Code]
1. 背景与底层痛点

研究动机： DPO (Direct Preference Optimization) 已经成为 RLHF 的主流替代方案，因为它无需训练 Reward Model。然而，DPO 强依赖于成对数据 (Paired Data, $y_w > y_l$)。在实际业务中，我们往往只有“点赞/点踩”的单点数据 (Unary Data)，这时候 KTO (Kahneman-Tversky Optimization) 被提出。但业界一直缺乏在千亿参数模型上对这三种方法的严格对比。

底层数学差异： PPO 优化的是 $E_{x,y \sim \pi}[r(x,y) - \beta \log \frac{\pi(y|x)}{\pi_{ref}(y|x)}]$。DPO 通过推导最优策略的封闭解，将损失函数转换为 $-\log \sigma (\beta \log \frac{\pi(y_w)}{\pi_{ref}(y_w)} - \beta \log \frac{\pi(y_l)}{\pi_{ref}(y_l)})$。KTO 则利用前景理论，分别对 $y_{good}$ 和 $y_{bad}$ 进行加权，彻底抛弃了 pair 结构。

3. 方法详解 (Methodology - Hardcore)

损失函数剖析：KTO 的非成对损失

KTO 的核心在于将人类的效用函数建模为参考点的偏差。损失函数如下：

$$ L_{KTO} = \mathbb{E}_{x, y \sim D} [w(y) (1 - \sigma(v_{KTO}(x, y; \beta)))] $$

其中 $v_{KTO}$ 是价值函数： $$ v_{KTO} = \begin{cases} \beta \log \frac{\pi(y|x)}{\pi_{ref}(y|x)} - z_{ref} & \text{if } y \text{ is desirable} \\ z_{ref} - \beta \log \frac{\pi(y|x)}{\pi_{ref}(y|x)} & \text{if } y \text{ is undesirable} \end{cases} $$

关键发现： 实验表明，DPO 在数据量较少时容易过拟合（Over-optimization），导致模型输出长度异常增加（Reward Hacking 的一种形式）。而 KTO 对超参数 $\beta$ 更不敏感，且在仅有 10% 成对数据的情况下，效果依然优于全量数据的 SFT。

🤖 LLM Agent

3. AgentCoder: Self-Reflective Multi-Agent Framework for Complex Code Generation

[arXiv PDF] [GitHub Code]
1. 背景与底层痛点

痛点： 现有的单体 Agent（如 AutoGPT）在处理复杂编程任务时，往往陷入“死循环”——写出 Bug 代码，运行报错，然后盲目修改，越改越错。这是因为缺乏一个独立的“测试工程师”角色来进行客观验证，且 LLM 自身的自我纠错能力在长 Context 下会显著退化。

3. 方法详解 (Methodology - Hardcore)

本文提出了一个标准的三角色博弈框架（Three-Role Game）：

  1. Programmer Agent: 负责根据需求写代码。
  2. Test Designer Agent: 这是一个创新点。它不写功能代码，而是专门根据需求生成 Unit Test Cases（单元测试用例）。
  3. Reviewer Agent: 运行测试用例，捕获 Traceback，并结合代码逻辑生成“自然语言修复建议”。

核心流程：Iterative Refinement Loop

while (tests_passed < threshold) and (step < max_steps): code = programmer.generate() tests = tester.generate() result = executor.run(code, tests) if result.success: break feedback = reviewer.analyze(code, result.error_log) # 关键点：将 feedback 注入到 programmer 的短期记忆中 programmer.memory.add(feedback)
4. 深度场景剖析与具体案例 (Examples)
Case Study: HumanEval #163 (复杂数据处理)
User: 编写一个函数，生成给定范围内的所有偶数数字序列，且数字本身必须是回文数。
Programmer (Round 1): ...生成了简单的 range(a, b) 检查... (Bug: 忽略了回文检查)
Tester: 生成测试用例 assert func(1, 100) == [2, 4, 6, 8, 11, 22...] (Wait, 11 is odd)
Reviewer: 报错！测试用例 11 期望通过，但你的代码输出了偶数。注意：题目要求“数字本身是偶数”还是“偶数个数字”？(Self-correction: 题目要求是 even digits)
Programmer (Round 2): 修正逻辑，增加 `if num % 2 == 0 and str(num) == str(num)[::-1]`... Success!

分析： 这里的关键是 Tester 甚至可能生成错误的测试用例，Reviewer 充当了仲裁者，通过多轮对话消除了对需求的歧义。

⚕️ 医学 LLM 与医疗 AI

4. Med-RAG-QA: Towards Trustworthy Medical QA via Hierarchical Knowledge Retrieval

[arXiv PDF] [GitHub Code]
1. 背景与底层痛点

根本缺陷： 通用 LLM (如 GPT-4) 在回答医学问题时常产生幻觉（Hallucination），例如编造药物剂量或副作用。直接外挂 Wikipedia 等通用知识库进行 RAG，由于医学术语的歧义性（如 "Cold" 既是感冒也是冷），检索精度极低（Precision@k < 40%）。医学指南通常是长文档，切片（Chunking）容易切断上下文逻辑。

3. 方法详解 (Methodology - Hardcore)

层级化检索 (Hierarchical Retrieval)

  • L1: 实体对齐 (Entity Linking): 使用 UMLS (Unified Medical Language System) 知识图谱，将用户 Query 中的非标准描述（"肚子疼"）映射为标准术语（"Abdominal Pain"）。
  • L2: 结构化指南检索: 不直接检索文本块，而是检索“决策树”。文章将临床指南（Clinical Guidelines）解析为流程图，检索的是图中的 Sub-graph。

Evidence-aware Generation

在 Prompt 中强制加入引用约束：

Answer the patient's question based strictly on the provided Context. For every claim, append a citation [ID]. If the context does not contain the answer, say "Insufficient clinical evidence".
5. 实验与多维数据 (Experiments)
Model Retrieval Source PubMedQA Acc MedQA (USMLE) Hallucination Rate (Lower is better)
GPT-4 Zero-shot None 75.2% 81.4% 12.5%
Llama-3-70B + Naive RAG Wikipedia 71.5% 76.0% 8.2%

结论： 本文方法将幻觉率压到了 2.1%，这对于医疗应用至关重要。虽然 Accuracy 提升看似不大，但错误类型的分布发生了质变——从“一本正经胡说八道”变成了“不知道就拒答”。

🔍 信息检索与医学检索

5. Dense Retrieval for EHR: Learning to Rank Patient Histories with Temporal Awareness

[arXiv PDF] [GitHub Code]
1. 背景与底层痛点

场景： 电子病历 (EHR) 包含大量非结构化文本（出院小结、病程记录）。当医生需要查找“该患者既往是否有过青霉素过敏史”时，传统的关键词搜索（BM25）经常失效，因为记录中可能写的是 "PCN allergy" 或 "allergic to Penicillin"。且病历具有强时间属性，10年前的记录权重应低于最近的记录。

3. 方法详解 (Methodology - Hardcore)

Time-aware Bi-Encoder Architecture

模型架构基于 BERT-base，但对 Embedding 进行了改造： $$ E_{final} = E_{text} + E_{segment} + E_{position} + E_{time} $$ 其中 $E_{time}$ 是通过正弦函数编码的时间间隔（Time Interval Embedding）。

Hard Negative Mining for Medical IR

训练时，如果不加区分地选取负样本，模型很容易通过“科室名称”（如心内科 vs 骨科）来作弊。本文提出了一种 **Clinical Hard Negative** 策略：强制选取同一科室、甚至同一诊断编码（ICD-10）但具体病情描述不匹配的文档作为负样本，迫使模型去理解具体的医疗语义，而不是利用元数据捷径。
