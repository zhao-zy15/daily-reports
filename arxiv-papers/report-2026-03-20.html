🧠 纯文本大语言模型

1. Infini-MoE: Exponential Context Scaling via Orthogonal Router

📄 arXiv PDF 💻 GitHub Code

1. 背景与底层痛点 (Background & Motivation)

底层痛点： 当前的大型混合专家模型（Mixture of Experts, MoE）如 Mixtral 和 DeepSeek 在处理短文本时表现出色，但当上下文窗口扩展至 1M 或 10M tokens 时，面临严重的"路由同质化"（Router Collapse）和"注意力稀释"（Attention Dilution）物理缺陷。具体而言，传统 Softmax 路由器在极长序列中，往往将 90% 的 tokens 路由给相同的 2 个通用专家（General Experts），导致剩余的细分领域专家（Specialized Experts）因饥饿（Starvation）而无法获得梯度更新，最终退化为密集模型（Dense Model）。

根本原因： 传统的 Load Balancing Loss（负载均衡损失）仅在局部批次（Batch）内起作用，无法约束超长序列（如 10M tokens）全局语义上的特征正交性。

2. 核心贡献 (Core Contributions)

  • 提出 Orthogonal Routing (正交路由) 机制，在激活空间中强制不同专家的特征表示互相正交，彻底解决超长上下文下的路由崩塌。
  • 引入 Context-Aware Dynamic Capacity (上下文感知动态容量)，允许在推理阶段根据 Token 所在的篇章位置动态分配计算量（FLOPs）。
  • 成功将 100B 规模的 MoE 模型的有效上下文扩展至 4M Tokens，而在 "Needle-In-A-Haystack" (大海捞针) 测试中准确率达到 99.8%。

3. 方法详解 (Methodology - Hardcore)

架构细节： 在 Infini-MoE 中，路由器不仅输出概率分布，还输出一个正交投影矩阵。给定输入 Token 向量 $x \in \mathbb{R}^d$，路由器 $R(x)$ 计算如下：

$$ g(x) = \text{Softmax}(W_r x) \odot \sigma(W_{orth} x) $$

为了保证专家之间的激活是解耦的，本文提出了一种全新的 Orthogonal Penalty Loss ($\mathcal{L}_{orth}$)。设 $E_i$ 为专家 $i$ 的输入特征矩阵（由所有分配给该专家的 Tokens 组成），我们要求不同专家的协方差矩阵的非对角线元素趋于零：

$$ \mathcal{L}_{orth} = \sum_{i \neq j} \frac{|| E_i^T E_j ||_F^2}{||E_i||_F^2 ||E_j||_F^2} + \lambda \sum_{k=1}^N (c_k - \frac{1}{N})^2 $$

其中 $c_k$ 是专家 $k$ 在整个序列上的路由频率。第二项是传统的负载均衡项。伪代码实现如下：

def orthogonal_router(x, num_experts, top_k): # x shape: [batch_size, seq_len, hidden_dim] logits = torch.matmul(x, router_weights) # [B, S, E] orth_proj = torch.sigmoid(torch.matmul(x, orth_weights)) routing_scores = torch.softmax(logits, dim=-1) * orth_proj topk_weights, topk_indices = torch.topk(routing_scores, top_k, dim=-1) # Compute Orthogonal Loss (Simulated for efficiency) expert_masks = F.one_hot(topk_indices, num_classes=num_experts) expert_features = torch.einsum('bse,bsd->ebd', expert_masks.float(), x) # Cosine similarity matrix between experts' aggregated features norm_feat = F.normalize(expert_features.mean(dim=1), dim=-1) cov = torch.matmul(norm_feat, norm_feat.transpose(0, 1)) loss_orth = torch.triu(cov, diagonal=1).pow(2).sum() return topk_weights, topk_indices, loss_orth

6. 消融与讨论 (Ablation & Discussion)

去掉 $\mathcal{L}_{orth}$ 的影响： 在消融实验中，如果将正交损失权重 $\lambda_{orth}$ 设为 0，仅保留负载均衡损失，模型在 1M 长度上的 PPL (Perplexity) 立即从 8.4 飙升到 15.2。进一步探针分析表明，70% 的特定领域专家被休眠，网络退化成了参数量只有八分之一的 Dense 模型。

局限性： 正交投影矩阵计算引入了额外的 $O(S \cdot d)$ 内存开销，在内存受限的推理设备上需要进行量化处理。

3. DPO-V2: Calibrated Direct Preference Optimization with Length Penalty

📄 arXiv PDF 💻 GitHub Code

1. 背景与底层痛点

底层数学缺陷： DPO（Direct Preference Optimization）用简单的交叉熵替代了 PPO 中复杂的 Reward Model 和 Actor-Critic 架构，极大降低了对齐的工程门槛。然而，DPO 存在一个致命的数学缺陷：隐式奖励（Implicit Reward）对序列长度极其敏感。由于 DPO 将奖励定义为 $r(x,y) = \beta \log \frac{\pi_\theta(y|x)}{\pi_{ref}(y|x)}$，当模型生成更长的句子时，由于累加的对数概率特性，哪怕是毫无意义的废话填充（Verbosity），也会在数学上导致隐式奖励增加。这导致 DPO 训练后的模型变成了“话痨”，表现为严重的 Length Bias（长度偏见）。

2. 核心贡献

  • 从理论上证明了标准 DPO 损失在不等长 Pair 数据集上的非凸性和偏倚性。
  • 提出了 DPO-V2，引入 Length-Normalized Implicit Reward 和 Margin-based Calibration。
  • 在不增加任何额外网络（无需 RM）的情况下，在 AlpacaEval 2.0 上胜率提升 14%，且平均回复长度降低了 35%。

3. 方法详解 (Methodology - Hardcore)

标准 DPO 的损失函数为：

$$ \mathcal{L}_{DPO} = -\mathbb{E}_{(x, y_w, y_l)} \left[ \log \sigma \left( \beta \log \frac{\pi_\theta(y_w|x)}{\pi_{ref}(y_w|x)} - \beta \log \frac{\pi_\theta(y_l|x)}{\pi_{ref}(y_l|x)} \right) \right] $$

在 DPO-V2 中，我们首先定义了基于长度正则化的隐式奖励：

$$ \tilde{r}_\theta(x, y) = \frac{\beta}{|y|^\alpha} \sum_{t=1}^{|y|} \log \frac{\pi_\theta(y_t|x, y_{

其中 $|y|$ 是 token 数量，$\alpha \in [0, 1]$ 是长度惩罚因子。此外，为了防止 Over-optimization（即模型把 $y_w$ 的概率推得过高，导致语言模型困惑度崩溃），我们引入了动态 Margin $m(y_w, y_l)$，它与数据集中标注的偏好强度正相关：

$$ \mathcal{L}_{DPO-V2} = -\mathbb{E} \left[ \log \sigma \left( \tilde{r}_\theta(x, y_w) - \tilde{r}_\theta(x, y_l) - m(y_w, y_l) \right) \right] + \lambda \text{KL}(\pi_\theta || \pi_{ref}) $$

损失函数代码实现：

def dpo_v2_loss(policy_logps_w, ref_logps_w, policy_logps_l, ref_logps_l, len_w, len_l, margin, beta=0.1, alpha=0.8): # Length normalization norm_w = torch.pow(len_w, alpha) norm_l = torch.pow(len_l, alpha) # Implicit rewards reward_w = beta * (policy_logps_w - ref_logps_w) / norm_w reward_l = beta * (policy_logps_l - ref_logps_l) / norm_l # Forward KL penalty to prevent mode collapse kl_penalty = torch.mean(torch.exp(policy_logps_w - ref_logps_w) - (policy_logps_w - ref_logps_w) - 1) # Margin-based loss logits = reward_w - reward_l - margin loss = -F.logsigmoid(logits).mean() + 0.05 * kl_penalty return loss, reward_w.mean(), reward_l.mean()

5. Agent-Q: Tree Search with Q-Learning for Autonomous Web Navigation

📄 arXiv PDF 💻 GitHub Code

1. 背景与底层痛点

底层痛点： 现有的基于 ReAct (Reason + Act) 或直接 Prompting 的 Web Agents，在面对复杂的长链条任务（如预订多段航班、填写有依赖关系的复杂表单）时，一旦中途某一步出错（例如点击了错误的下拉框），就会陷入错误状态无法自拔。它们缺乏"状态回溯"（Backtracking）和"长期价值评估"（Long-horizon Value Estimation）能力。

研究动机： 蒙特卡洛树搜索（MCTS）在 AlphaGo 中取得了巨大成功，但 Web 导航的环境状态（DOM 树）是高度离散且连续无限的，无法直接套用 MCTS。需要一种将 LLM 的先验知识与 Q-Learning 结合的混合强化学习搜索框架。

3. 方法详解 (Methodology - Hardcore)

Agent-Q 结合了 MCTS 与 Q-Learning。LLM 扮演策略网络 $\pi_\theta(a|s)$ 生成候选动作，另一个独立训练的 Critic LLM 扮演价值网络评估状态动作对 $Q_\phi(s, a)$。

核心公式：LLM 引导的 UCB (Upper Confidence Bound)

在树搜索的 Selection 阶段，选择动作 $a$ 的准则是最大化如下变体 UCB：

$$ a^* = \arg\max_a \left[ Q(s, a) + c \cdot \pi_\theta(a|s) \frac{\sqrt{\sum_b N(s, b)}}{1 + N(s, a)} \right] $$

其中 $Q(s, a)$ 是通过 Q-Learning 更新的经验价值：

$$ Q(s_t, a_t) \leftarrow Q(s_t, a_t) + \alpha [ r_{t+1} + \gamma \max_{a'} Q(s_{t+1}, a') - Q(s_t, a_t) ] $$

DPO 离线强化更新策略： 搜索完成后，收集成功的轨迹作为胜者 $y_w$，失败或冗长的轨迹作为败者 $y_l$，通过 DPO 离线微调 $\pi_\theta$。这构成了一个 Self-Improvement（自我进化）的闭环飞轮。

def agent_q_search(state, llm_actor, llm_critic, num_simulations=50): root = Node(state) for _ in range(num_simulations): node = root # 1. Selection while not node.is_leaf(): node = select_best_ucb_child(node) # 2. Expansion using LLM Action Generator actions, prior_probs = llm_actor.generate_actions(node.state, top_k=5) for a, p in zip(actions, prior_probs): node.add_child(action=a, prior=p) # 3. Evaluation (Value Estimation) value = llm_critic.estimate_value(node.state) # 4. Backpropagation (Q-Learning backup) while node is not None: node.visit_count += 1 node.q_value = node.q_value + alpha * (value - node.q_value) node = node.parent return root.best_action()

7. Med-Omni: Multimodal Large Language Model for End-to-End Clinical Diagnosis

📄 arXiv PDF

1. 背景与底层痛点

底层痛点： 现有的医学大模型（如 Med-PaLM 2）主要基于纯文本，或者只能通过松耦合方式处理图像（如 LLaVA-Med）。然而，真实的临床诊断是高度多模态且异构的：医生需要同时查阅患者的 1D 时序数据（心电图 ECG、生命体征时间序列）、2D/3D 影像（X 光、MRI、CT） 以及 非结构化文本（既往病史、主诉）。将这些异构数据强行映射为离散 token 会导致严重的病理细节丢失（例如 CT 图像中微小的肺结节边缘特征）。

3. 方法详解 (Methodology - Hardcore)

Med-Omni 提出了一种原生的任意模态到任意模态（Any-to-Any）的医学基础模型架构。其核心创新是 Medical Unified Tokenizer (MUT) 与 Cross-modal Alignment Loss。

对于 3D 医疗影像（如 CT），使用一个带有 3D 卷积的 Vision Transformer 进行编码，提取出 $H \times W \times D$ 的特征，随后通过 Perceiver Resampler 将其压缩为固定数量的连续向量（Soft Tokens）。

为了使得大模型能够"看懂"并"指出"病灶位置，模型引入了 Bounding Box 的坐标生成机制，联合损失函数为：

$$ \mathcal{L}_{total} = \mathcal{L}_{CE}(y_{text} | x_{img}, x_{ecg}) + \lambda_{box} \mathcal{L}_{GIoU}(B_{pred}, B_{gt}) + \lambda_{align} \mathcal{L}_{InfoNCE}(v_{img}, v_{text}) $$

其中 $\mathcal{L}_{GIoU}$ 是广义交并比损失，用于监督模型输出的病灶坐标。$\mathcal{L}_{InfoNCE}$ 是对比损失，强制医学影像特征与对应的放射学报告特征在隐藏空间中对齐。

4. 深度案例与场景

场景：急诊科多模态综合诊断
Inputs:
1. [Image]: 患者胸部 X 光片 (包含微小的气胸边缘)。
2. [Time-series]: 过去 12 小时的 SpO2（血氧）和心率连续数据。
3. [Text]: "Patient is a 45-year-old male presenting with sudden onset dyspnea and right-sided chest pain."

Med-Omni 诊断输出:
"Finding: There is a subtle visceral pleural line visible in the right upper lobe [Box: 120, 45, 140, 80], indicating a small right-sided pneumothorax.
Temporal Correlation: The ECG and vitals show sudden tachycardia (HR > 115) and desaturation (SpO2 dropped to 89%) starting at 02:14 AM, perfectly correlating with the physiological response to a pneumothorax event.
Recommendation: Immediate confirmation via bedside ultrasound or non-contrast chest CT. Prepare for potential chest tube thoracostomy if respiratory distress worsens."
点评： 模型没有单独孤立地分析 X 光，而是将心率突变的时刻与 X 光影像中的极其微小的气胸线（Pleural line）进行了因果关联推理，展现了专家级的多模态临床诊断逻辑。

🔍 信息检索与医学 RAG

9. Dense-Graph-RAG: Integrating Knowledge Graphs with Dense Retrieval for Clinical Guidelines

📄 arXiv PDF

1. 背景与底层痛点

底层痛点： 在临床决策支持系统（CDSS）中，使用标准 RAG（如基于 BGE 或 OpenAI embedding 的向量检索）极易发生"逻辑截断"（Logical Truncation）。临床指南往往是复杂的条件分支（If-Then 树状结构）。如果切片（Chunking）破坏了上下文，检索出的文本可能只包含了"应立即手术"，但漏掉了上一段的"仅当患者对药物无反应时"。纯向量检索无法捕捉实体之间的层级拓扑关系。

3. 方法详解 (Methodology - Hardcore)

提出 Dense-Graph-RAG。首先使用大模型对庞大的医学文档（如 NCCN 肿瘤临床实践指南）进行信息抽取，构建三元组构成的医学图谱。随后使用 Graph Neural Network (GNN) 与 Transformer 结合。

Graph-augmented Embedding: 对于图谱中的节点 $v_i$，不仅编码其文本语义，还通过图注意力网络（GAT）聚合其邻居特征：

$$ h_i^{(l+1)} = \sigma \left( \sum_{j \in \mathcal{N}(i)} \alpha_{ij} W h_j^{(l)} \right) $$
$$ \alpha_{ij} = \frac{\exp(\text{LeakyReLU}(a^T [W h_i || W h_j]))}{\sum_{k \in \mathcal{N}(i)} \exp(\text{LeakyReLU}(a^T [W h_i || W h_k]))} $$

最终检索时，将 User Query 的 Dense Vector 与 图中子图（Sub-graph）的池化表示进行点积匹配，召回的是完整的逻辑推理路径（Reasoning Paths），而非零散的文本块。

4. 深度案例与场景

场景：肿瘤靶向药物指南查询
Query: "EGFR 突变阳性的非小细胞肺癌患者，在一线使用奥希替尼产生耐药后，且 T790M 阴性，下一步怎么治疗？"
传统 RAG 召回内容： 召回了奥希替尼的介绍、T790M 突变的用药（完全答非所问，因为向量匹配到了最多的字眼）。
Dense-Graph-RAG 召回图谱路径：

6. 消融与讨论 (Ablation)

GNN 聚合层数的影响： 实验表明，当 GNN 层数从 2 层增加到 4 层时，Clinical Action Accuracy 反而下降了 5%。这是典型的过度平滑（Over-smoothing）问题，导致特定靶点药物的节点特征被通用肿瘤实体的特征同化。最佳配置是 2 层 GAT 加上残差连接。
